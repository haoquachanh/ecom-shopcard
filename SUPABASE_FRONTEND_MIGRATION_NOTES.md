# Supabase Frontend Migration Notes

## Frontend env

`frontend` public deploy only needs:

```env
VITE_SUPABASE_URL=
VITE_SUPABASE_ANON_KEY=
```

Do not add `DATABASE_URL` or `SUPABASE_SERVICE_ROLE_KEY` to `frontend`.

## Tables Read By Frontend

- `product_types`
- `samples`
- `materials`
- `sizes`
- `sides`
- `effects`
- `base_prices`
- `quantity_tiers`

## Backend API Reads Removed From Public Frontend

- `GET /product-types`
- `GET /product-types/:id`
- `GET /product-types/slug/:slug`
- `GET /samples`
- `GET /samples/:id`
- `GET /samples/slug/:slug`
- `GET /price-grids/tiers`
- `GET /price-grids`
- `GET /price-grids/extra-charges`
- `POST /price-grids/calculate`

Admin/auth frontend files were removed from `frontend` public routing. Use the separate `admin/` app and backend for admin mutation workflows.

## Schema Gaps To Confirm

The current database schema does not have these frontend concepts:

- Product/sample `slug`: frontend now derives route slugs from `name` + `id`, for example `standee-mica-12`.
- Sample `tags`: frontend maps this to an empty array.
- Sample gallery/images: frontend uses `samples.image_url` as the primary/thumbnail image.
- `banners` or home sections: no matching table exists; `banners.service.ts` returns an empty list.
- Legacy price-grid fields (`price_tiers`, `extra_charges`, JSON grid data): frontend now reads `base_prices` and `quantity_tiers`.

If these features are required, add explicit columns/tables and update the generated frontend database types.

## Required RLS Policies

Run this in Supabase SQL Editor after creating the schema. These policies allow anon users to read public catalog/pricing data only. They do not allow frontend INSERT/UPDATE/DELETE.

```sql
alter table product_types enable row level security;
alter table samples enable row level security;
alter table materials enable row level security;
alter table sizes enable row level security;
alter table sides enable row level security;
alter table effects enable row level security;
alter table base_prices enable row level security;
alter table quantity_tiers enable row level security;

create policy "Public active product types"
on product_types for select
to anon
using (is_active = true);

create policy "Public active samples"
on samples for select
to anon
using (
  is_active = true
  and exists (
    select 1
    from product_types
    where product_types.id = samples.product_type_id
      and product_types.is_active = true
  )
);

create policy "Public active materials"
on materials for select
to anon
using (is_active = true);

create policy "Public active sizes"
on sizes for select
to anon
using (is_active = true);

create policy "Public sides"
on sides for select
to anon
using (true);

create policy "Public active effects"
on effects for select
to anon
using (is_active = true);

create policy "Public base prices"
on base_prices for select
to anon
using (
  product_type_id is null
  or exists (
    select 1
    from product_types
    where product_types.id = base_prices.product_type_id
      and product_types.is_active = true
  )
);

create policy "Public quantity tiers"
on quantity_tiers for select
to anon
using (
  exists (
    select 1
    from base_prices
    left join product_types on product_types.id = base_prices.product_type_id
    where base_prices.id = quantity_tiers.base_price_id
      and (base_prices.product_type_id is null or product_types.is_active = true)
  )
);
```

For Storage product images, use a public bucket for public catalog images, or signed URLs generated by backend/admin workflows if the bucket must stay private.
